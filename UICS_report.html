1. Abstract

This project introduces an ontology based on OWL to articulate interactive input/output controls and components within UI component APIs, leveraging popular framework Ionic UI Components, particularly in the context of Angular applications. The developed knowledge model employs controlled language constructs, offering the means to specify design intentions using keywords. It is designed to intelligently generate web interface controls through a SPARQL endpoint, facilitating the creation of sections with defined roles.

2. Introduction

In web development, the creation of interactive and attractive user interfaces relies on UI component libraries and frameworks. This projects addresses the challenge of describing a design and generating corresponding interface controls using library Ionic UI Components for Angular.

The motivation cone from the need to provide designers a more semantically rich means of specifying web elements. To achieve this, we propose the development of an ontology grounded in OWL , offering a structured knowledge model. This model, using controlled language constructs, enables the utilization of descriptive keywords and parameters to express design requirements. By utilizing a SPARQL endpoint, the ontology aims to intelligently interpret these specifications and generate appropriate controls.

Through this project, we present a practical validation through the implementation and testing of the proposed ontology in at least two distinct use cases.

3. Application Overview

3.1. Application Architecture

The architecture of UICS is delineated into distinct layers, each serving an important role in the overall functionality of the system.

Frontend

The Frontend component of UICS is responsible for presenting the user interface, facilitating user interaction, and collecting the controlled language input. It provides a user-friendly method for users to articulate their design intentions.

Backend

The Backend layer's main purpose is to process the user's controlled language input and generating Ionic UI Components-based code accordingly. It serves as the computational engine that transforms abstract design descriptions into interface elements.

Database

UICS leverages two distinct storage systems to manage its data effectively.

RDF Storage System (Apache Jena Fuseki): The RDF storage system is used for persistently storing and managing Resource Description Framework (RDF) data, namely the ontology we create to express the relations between Ionic UI components. Apache Jena Fuseki serves as the database engine.

Cache Storage (MySQL): In addition to the RDF storage, UICS utilizes a MySQL-based Cache Storage system. This system provides a quick and efficient means of storing and retrieving frequently accessed data, optimizing the performance of the backend processes.

3.2. Frontend Architecture

This architectural design ensures a simple, user-friendly experience, allowing individuals to describe their webpage visions effortlessly. The architecture of the system is designed with a separation between the User Interface (UI) and the Backend, encouraging modularity. The system empowers users to articulate their webpage design intentions through a controlled language, interact with the resulting code, and access information about the language conventions.

Main Page

The main page of the User Interface serves as the central hub for users to express their webpage design specifications. It comprises the following components:

  • Input Text Area

    Users can describe the structure and content of the webpage they want to create using natural language. This controlled language facilitates communication between the user and the system, enabling a effortless design process.

  • Output Area

    The system dynamically generates and displays the resulting code based on the user's input. This allows users to view and understand the code corresponding to their design description.

  • Info Link

    A link is provided to an information page that explains the conventions of the controlled language. This page serves as a guide, helping users create precise descriptions for their webpage elements.

Info Page

The Info Page offers comprehensive guidance on the controlled language that users should employ to describe their webpage. It covers keywords, and parameters, providing users with the necessary information to communicate their design intentions.

3.3. Backend Architecture

3.3.1 Architecture

The backend architecture of the system is designed to handle HTTP requests, execute conversions, and manage data in multiple databases. Components are interlinked to ensure a smooth flow of data.

HTTP Server

The HTTP server acts as the central hub for handling and routing HTTP requests. It is implemented using Python and the Flask framework, providing a robust foundation for web server functionalities.

Request Handlers

  • GET Handler

    The GET handler is responsible for retrieving already executed conversions based on the provided "conversionID" parameter. It is in direct relation with the HTTP server, making HTTP requests to fetch the required data from Cache Storage - Database.

  • DELETE Handler

    The DELETE handler facilitates the removal of previously executed conversions, using the "conversionID" parameter. Like the GET handler, it directly communicates with the HTTP server through HTTP requests and removes data from Cache Storage - Database.

  • POST Handler

    The POST handler creates new conversions based on user input. It is connected to the Controlled Language Converter, Query Executor, and Query Result Adapter, forming a pipeline for conversion creation. After conversion is created, it saves data to Cache Storage - Database.

Conversion Processing

  • Controlled Language Converter

    The Controlled Language Converter interprets user descriptions, expressed in a controlled language, and converts them into SPARQL queries. It collaborates with the POST handler to process user input effectively.

  • Query Executor

    The Query Executor receives SPARQL queries generated by the Controlled Language Converter and executes them. It is directly linked to the second database (RDF Storage System Fuseki) to fetch data using pyfuseki library.

  • Query Result Adapter

    The Query Result Adapter processes the results of SPARQL queries from the Query Executor, transforming them into user-friendly code. It works with the POST handler to present the converted data to users.

3.3.2. API Specification

This REST API specification provides a standardized and accessible interface for users to interact with the UI Component Smart Generator. It outlines clear paths for listing, creating, retrieving, and deleting conversions based on user-defined design descriptions.

4. User Interaction

The User plays the main role in the system, initiating the interface design process by describing their vision in a controlled language. This controlled language acts as a structured way for users to describe the webpage layout. The User can access the Info Page from the Main Page to find out the structure and keywords of the controlled language.

UICS acts as the intermediary that interprets and transforms the user's controlled language input into code, specifically utilizing Ionic UI Components. After conversion, the code is returned to the User on an output area from Main Page for display.

5. Ontology

The OWL-based ontology developed for Ionic UI components serves as a foundational framework for expressing and formalizing the semantics of key elements within the Ionic Framework. This section provides an overview of the key components.

Within the ontology, each Ionic UI component is modeled as a class, forming a structured hierarchy that reflects their organizational relationships. For instance, the class structure represents the parent-child relationship between ion-card and ion-card-content. This hierarchical structure facilitates efficient navigation and retrieval of information about the components and their interdependencies.

List of Ionic UI components used in UICS project

  • ion-input

    The input component is a wrapper to the HTML input element with custom styling and additional functionality. It accepts most of the same properties as the HTML input, but works great on desktop devices and integrates with the keyboard on mobile devices.

  • ion-content

    The content component provides an easy-to-use content area with some useful methods to control the scrollable area. There should only be one content in a single view.

  • ion-header

    Header is a root component of a page that aligns itself to the top of the page. It is recommended to be used as a wrapper for one or more toolbars, but it can be used to wrap any element. When a toolbar is used inside a header, the content will be adjusted so it is sized correctly, and the header will account for any device safe areas.

  • ion-footer

    Footer is a root component of a page that aligns itself to the bottom of the page. It is recommended to be used as a wrapper for one or more toolbars, but it can be used to wrap any element. When a toolbar is used inside a footer, the content will be adjusted so it is sized correctly, and the footer will account for any device safe areas.

  • ion-card

    Cards are containers that display content such as text, images, buttons, and lists. A card can be a single component, but is often made up of a header, title, subtitle, and content. Contains:

    • ion-card-content

      Card content is a child component of card that adds padding around its contents. It is recommended that any text content for a card should be placed inside of card content.

    • ion-card-header

      Card header is a child component of card that should be placed before the card content. It can contain a card title and a card subtitle.

    • ion-card-title

      Card title is a child component of card that should be placed inside a card header.

    • ion-card-subtitle

      Card subtitle is a child component of card that should be placed inside a card header.

  • ion-grid

    The grid is a powerful mobile-first flexbox system for building custom layouts. It is composed of three units — a grid, row(s) and column(s). Columns will expand to fill the row, and will resize to fit additional columns. It is based on a 12 column layout with different breakpoints based on the screen size. The number of columns can be customized using CSS. Contains:

    • ion-row

      Rows are horizontal components of the grid system and contain varying numbers of columns. They ensure the columns are positioned properly. Contains:

      • ion-col

        Columns are cellular components of the grid system and go inside a row. They will expand to fill the row. All content within a grid should go inside a column.

  • ion-list

    Lists are made up of multiple rows of items which can contain text, buttons, toggles, icons, thumbnails, and much more. Lists generally contain items with similar data content, such as images and text. Contains:

    • ion-item

      Items are elements that can contain text, icons, avatars, images, inputs, and any other native or custom elements. Items should only be used as rows in a List with other items. Items can be swiped, deleted, reordered, edited, and more.
